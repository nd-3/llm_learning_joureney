# ID19 原因調査

対象: `evals/qa_dataset_v2.json` ID19(両モデル共通の不正解)。本ノートは原因調査のみを行い、修正は行っていません。

## 1. 対象設問

```json
{
  "id": 19,
  "type": "multi",
  "question": "関数のローカル変数とクラスのインスタンス変数は、それぞれどのように扱われますか?",
  "reference_answer": "関数内で定義されたローカル変数は関数の実行が終了すると失われるのに対し、self.を付けて宣言するインスタンス変数はそのオブジェクトが使われている間、値を保持し続けます。",
  "expected_keywords": ["ローカル変数", "self", "保持"]
}
```

`multi`問題であり、模範解答は「関数のローカル変数」の章(第9章)と「クラスのインスタンス変数」の章(第13章)、2箇所の記述を組み合わせる必要がある設計です。

## 2. 根拠チャンクの特定

`src/main.py`と同じ設定(`chunk_size=1000`, `chunk_overlap=200`)でPDF全体を分割すると376チャンクになります。この中から、設問の根拠になる記述を含むチャンクを特定しました。

| チャンク番号 | PDFページ(0始まり) | 章 | 内容 |
|---|---|---|---|
| **176** | 123(表示上p.124) | 9.8 関数内の変数の扱い | ローカル変数の正式な定義 |
| **264** | 179(表示上p.180) | 13.3 末尾 | self.の有無でローカル変数とインスタンス変数を対比 |
| **267** | 181(表示上p.182) | 13.4 クラスの変数とアクセスの制限 | インスタンス変数の生成・動作(保持)・アクセスの説明 |

### チャンク176(ch9.8、ローカル変数の正式な定義)

```

Python では関数内の変数は以下のように扱います．コラム「変数のスコープ」も参
照してください．
 ローカル変数： 関数内で定義された（代入された）変数は関数内でのみ利用可
能で， 関数の（毎回の）実行ごとに関数の実行が終了すれば失われます．
 仮引数もローカル変数として扱われます．
 グローバル変数： 関数外で定義されている変数は値を読み取ることのみ可能で
す．
...
```

### チャンク264(ch13.3末尾、self.の有無による対比)

```
 インスタンス変数と初期化（ 3～６行）．__init__() メソッド内で行っているの
はクラスで使う変数 の初期化です．
 self. で始まる変数は「インスタンス変数」と呼ばれ， そのクラスのオブジ
ェクトが生成されるごとにオブジェクト固有で， オブジェクトの中では永続
的に使える変数です ．
 これに対し， self. を付けない変数は ，関数と同様ローカル変数として扱わ
れ，メソッドの処理が終わると捨てられます．
```

### チャンク267(ch13.4、インスタンス変数の生成・動作・アクセス)

```
 インスタンス変数
 生成：メソッド の定義の中で  self. を前につけて宣言します ．
 動作：生成されたインスタンスごとに独立した変数として扱われます．その
インスタンスが使われている間は値を保持します．
 アクセス：メソッドの定義の中では生成と同様 self. を前につけて参照しま
す．
...
```

`expected_keywords`(`ローカル変数`/`self`/`保持`)との対応は以下の通りです。

- `ローカル変数`: チャンク176・264の両方に含まれる
- `self`: チャンク264・267に含まれる
- `保持`: **チャンク267にのみ**含まれる(「値を保持します」)

つまり、キーワードを3つとも回答に含めるには、少なくともチャンク264(または176)とチャンク267の**組み合わせ**が必要です。

## 4. top_k=20まで広げた場合の順位

同じクエリで`k=20`まで取得し、根拠チャンクの順位を確認しました。

| 根拠チャンク | 内容 | top_k=20での順位 | score |
|---|---|---|---|
| 267 | インスタンス変数の生成・動作(保持)・アクセス | **7位** | 1.0789 |
| 264 | self.の有無によるローカル/インスタンス変数の対比 | **8位** | 1.1408 |
| 176 | ch9.8 ローカル変数の正式な定義 | **17位** | 1.2482 |

- キーワード`保持`の根拠となるチャンク267は7位、`self`+`ローカル変数`の対比を含むチャンク264は8位。**この2つは`top_k`を8程度まで広げれば両方とも回収できます。**
- 一方、ch9.8のフォーマルな定義(チャンク176)は17位までズレ込んでおり、拾うには`top_k`をかなり大きく(17以上)する必要があります。ただし、チャンク264だけでも「ローカル変数」の概念は(やや簡略化された形で)説明されているため、回答の完成には必須ではなさそうです。

## 5. 診断

**該当: (a) 根拠チャンクは存在するが順位が低い(top_k・クエリ表現の問題)**

根拠となるチャンクはベクトルDB内に明確に存在し、しかもチャンク264は「ローカル変数」と「インスタンス変数」を1チャンク内で対比的に説明しているため、チャンク分割自体が説明を不自然に泣き別れさせているわけではありません((b)には該当しません)。問題は、質問文の埋め込みに対してこれらのチャンクの類似度スコアが伸びず、現在の`top_k=4`の外(7位・8位)に押し出されていることです。

(c)「複数チャンクの組み合わせが必要」という側面も部分的にはあります。キーワード`保持`はチャンク267にしかなく、チャンク264だけでは`expected_keywords`を満たせません。ただし、この2チャンクは順位で言えば隣接した7位・8位であり、`top_k`を適度に引き上げるだけで両方とも自然に回収できる位置にあります。したがって根本原因は(a)であり、(c)はその結果として付随する事情と捉えるのが妥当です。

チャンク176(ch9.8のフォーマルな定義)は17位まで下がっており、これだけを回収しようとすると`top_k`をかなり大きくする必要があります。ただし4で述べた通り、チャンク264がローカル変数についても最低限の説明を含んでいるため、正解に必須ではない可能性があります。

## まとめ

- 根拠は本文中に実在する(架空の期待ではない)
- 根本原因は**top_k不足**(現行4に対し、決定的な2チャンクは7位・8位)
- チャンク分割の泣き別れ(境界の悪さ)が主因とは言えない
- 次のアクションとしては、`top_k`を7〜8程度まで引き上げた場合にID19が改善するかを試すのが妥当な検証ステップ(本ノートでは未実施)
